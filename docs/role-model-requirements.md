# Технические требования: ролевая модель `customer`, `operator`, `admin`

## Назначение

Фича заменяет старую модель доступа на основе булевого признака `is_superadmin` на единую ролевую модель с тремя ролями:

- `customer` - обычный пользователь;
- `operator` - операционный сотрудник;
- `admin` - администратор системы.

Цель - разделить пользовательские, операционные и административные сценарии, а также убрать зависимость сервисов и интерфейса от старых superadmin API.

## Роли

### `customer`

Обычный авторизованный пользователь.

Должен иметь доступ к:

- просмотру каталога продуктов;
- просмотру поставщиков;
- просмотру складов и товаров на складах;
- созданию своих заказов;
- просмотру своих заказов;
- скачиванию документов только по своим заказам;
- участию в своих чатах;
- просмотру и скачиванию документов поставщиков.

Не должен иметь доступ к:

- созданию, редактированию и удалению продуктов;
- созданию, редактированию и удалению поставщиков;
- созданию, редактированию и удалению складов;
- изменению складских остатков;
- approval queue;
- просмотру всех заказов;
- управлению чатами;
- управлению пользователями и ролями.

### `operator`

Операционный пользователь.

Должен иметь все возможности `customer`, а также:

- создавать, редактировать и удалять продукты;
- загружать изображения продуктов;
- создавать, редактировать и удалять поставщиков;
- загружать и удалять документы поставщиков;
- создавать, редактировать и удалять склады;
- добавлять, изменять и удалять товары на складах;
- работать с approval queue;
- просматривать все заказы;
- скачивать PDF-документы по любому заказу;
- менять операционные статусы заказов;
- создавать чаты и добавлять участников;
- читать чаты вне собственного участия для операционной поддержки.

Не должен иметь доступ к:

- изменению пользователей;
- удалению пользователей;
- назначению ролей.

### `admin`

Администратор системы.

Должен иметь все возможности `operator`, а также:

- просматривать полный список пользователей;
- редактировать пользователей;
- удалять пользователей;
- назначать роли `customer`, `operator`, `admin`;
- открывать административные страницы управления пользователями.

## Ролевая иерархия

Роли имеют порядок доступа:

```text
customer < operator < admin
```

Если endpoint требует `operator`, то доступ разрешен ролям `operator` и `admin`.

Если endpoint требует `admin`, то доступ разрешен только роли `admin`.

## Auth API

### Проверка токена

`POST /verify-token` должен:

- декодировать access token;
- найти пользователя в БД;
- вернуть `valid`, `user_id`, `role`.

Успешный ответ:

```json
{
  "valid": true,
  "user_id": "uuid",
  "role": "customer"
}
```

Если токен невалиден или пользователь не найден, ответ должен иметь:

```json
{
  "valid": false,
  "error": "..."
}
```

### Текущий пользователь

`GET /me` должен возвращать данные текущего пользователя:

```json
{
  "id": "uuid",
  "name": "User Name",
  "email": "user@example.com",
  "role": "customer"
}
```

### Назначение роли

`PUT /users/{user_id}/role` должен:

- быть доступен только `admin`;
- принимать тело `{ "role": "customer|operator|admin" }`;
- обновлять роль пользователя;
- запрещать снятие роли `admin` с последнего администратора.

## Удаленные контракты

Приложение не должно использовать:

- `GET /check-superadmin`;
- `POST /verify-token-with-admin`;
- `PUT /users/promote/{user_id}`;
- клиентские проверки `is_superadmin` / `isSuperAdmin`.

## UI требования

- Меню должно показывать раздел "Список пользователей" только `admin`.
- Меню должно показывать раздел "На согласование" только `operator` и `admin`.
- Кнопка перехода в админку заказов должна быть видна только `operator` и `admin`.
- Операционные кнопки создания, редактирования и удаления продуктов, поставщиков, складов и складских остатков должны быть скрыты для `customer`.
- Список пользователей должен показывать роли:
  - `customer` -> "Клиент";
  - `operator` -> "Оператор";
  - `admin` -> "Администратор".
- В списке пользователей должен быть доступен выбор роли вместо старой кнопки повышения до superadmin.
- Сервер должен защищать operator/admin HTML-страницы, даже если пользователь вручную откроет URL.

## Миграция данных

При старте `auth_service` система должна:

- добавить колонку `role`, если ее еще нет;
- перенести старые данные:
  - `is_superadmin = true` -> `role = 'admin'`;
  - остальные пользователи -> `role = 'customer'`;
- нормализовать невалидные роли в `customer`;
- назначить seed-пользователю роль `admin`.

## Acceptance Criteria

- Новый пользователь после регистрации получает роль `customer`.
- Seed admin создается или обновляется с ролью `admin`.
- `POST /verify-token` возвращает реальную роль пользователя.
- `GET /me` возвращает текущего пользователя и роль.
- `customer` не может выполнять write-операции продуктов, поставщиков, складов и остатков.
- `operator` может выполнять операционные write-операции, но не может менять роли пользователей.
- `admin` может менять роли пользователей.
- Нельзя удалить или понизить последнего `admin`.
- `customer` не видит чужой заказ.
- `operator` и `admin` видят чужие заказы и могут скачивать PDF по ним.
- `customer` не может читать чужой чат через `GET /chats/{chat_id}`.
- В коде не остается вызовов старых superadmin endpoint-ов.
