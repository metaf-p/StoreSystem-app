# Технические требования: документы поставщика и PDF-документы заказа

## Назначение

Фича добавляет два независимых бизнес-сценария:

- загрузка и хранение документов поставщика;
- выгрузка PDF-документов по заказу и отгрузке.

Цель - закрыть документооборот вокруг справочника поставщиков и операционного процесса заказа без изменения существующей модели оформления заказа.

## Роли и доступ

- Авторизованный пользователь:
  - просматривает список документов поставщика;
  - скачивает документы поставщика;
  - скачивает PDF-документы только по своим заказам.
- Superadmin:
  - загружает документы поставщика;
  - удаляет документы поставщика;
  - просматривает и скачивает документы поставщика;
  - скачивает PDF-документы по любым заказам.

Все запросы к защищенным операциям должны передавать:

```http
Authorization: Bearer <access_token>
```

## Документы поставщика

### Функциональные требования

- В разделе "Поставщики" должна быть доступна работа с документами поставщика.
- Для каждого поставщика должна открываться секция или модальное окно "Документы".
- Superadmin должен иметь возможность загрузить файл и указать:
  - тип документа;
  - файл;
  - необязательное описание.
- Любой авторизованный пользователь должен видеть список документов поставщика.
- Любой авторизованный пользователь должен скачивать документ поставщика.
- Только superadmin должен удалять документ поставщика.
- При удалении поставщика система должна запрещать удаление, если у поставщика есть связанные документы.

### Типы документов

Поддерживаемые значения `document_type`:

- `contract` - договор;
- `certificate` - сертификат;
- `requisites` - реквизиты;
- `price_list` - прайс-лист;
- `other` - другое.

### Валидация

- Максимальный размер файла: 10 MB.
- Система принимает любые типы файлов.
- Загруженный файл не должен сохраняться под исходным именем.
- Исходное имя файла хранится только как метаданные.
- Файл должен сохраняться под UUID-именем.
- Невалидный `document_type` должен возвращать ошибку `422`.
- Несуществующий `supplier_id` должен возвращать ошибку `404`.
- Несуществующий `document_id` должен возвращать ошибку `404`.

## PDF-документы заказа и отгрузки

### Функциональные требования

- В разделах "Мои отгрузки" и "Заказы" должны быть кнопки:
  - "Скачать счет";
  - "Скачать отгрузочный документ".
- Пользователь должен скачивать PDF только по своим заказам.
- Superadmin должен скачивать PDF по любому заказу.
- PDF должен генерироваться на лету по данным заказа.
- Сохранять сгенерированный PDF в БД или на диске не требуется.

### Содержимое PDF

PDF должен включать:

- номер заказа;
- ID пользователя;
- статус заказа;
- дату создания заказа;
- список позиций заказа;
- ID товара;
- ID склада;
- количество;
- цену;
- итог по строке;
- общую сумму.

### Ошибки

- Несуществующий заказ должен возвращать `404`.
- Попытка скачать чужой заказ без прав superadmin должна возвращать `403`.
- Неавторизованный запрос должен возвращать `401` или `403` в зависимости от точки отказа авторизации.

## Нефункциональные требования

- Документы поставщика должны храниться в persistent volume контейнера `products_service`.
- Метаданные документов должны храниться в БД `products_service`.
- PDF должен генерироваться в `orders_service`.
- UI должен скачивать файлы через `fetch` с Bearer token, затем сохранять blob как файл.
- Интерфейс не должен показывать форму загрузки и кнопки удаления документов пользователям без прав superadmin.

## Acceptance Criteria

- Superadmin загружает документ поставщика и видит его в списке документов.
- Обычный пользователь видит и скачивает документ поставщика.
- Обычный пользователь не видит форму загрузки и не может удалить документ.
- API запрещает upload/delete документа обычному пользователю.
- Поставщик с документами не удаляется.
- Пользователь скачивает счет и отгрузочный PDF по своему заказу.
- Пользователь не скачивает PDF чужого заказа.
- Superadmin скачивает PDF любого заказа.
- PDF response имеет `Content-Type: application/pdf`.
